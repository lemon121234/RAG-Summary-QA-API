## 🤖 RAG 專案介紹

### 開場（30秒）

> 「我開發了一個 RAG（檢索增強生成）系統，使用 FastAPI 和 Ollama 實現。這個系統可以上傳文檔，建立向量索引，然後根據知識庫回答問題。我將原本 800+ 行的單一文件重構為分層架構，現在主文件只有 125 行，每個層都有明確的職責。」

### 技術架構（1-2分鐘）

> 「我採用五層架構設計：
> 
> 1. **ingest 層**：負責文本切割和向量嵌入。我實作了可配置的 chunking strategy，支持調整 chunk size 和 overlap。
> 
> 2. **vectorstore 層**：管理向量資料庫。我設計了抽象層，如果要替換為 ChromaDB 或 Pinecone，只需要修改這一個文件。
> 
> 3. **retriever 層**：使用 cosine similarity 進行相似度搜索，返回 top-k 最相關的片段。
> 
> 4. **llm 層**：構建 prompt 並調用 LLM 生成答案，同時計算信心程度。
> 
> 5. **routes 層**：提供 RESTful API 端點。
> 
> 這種架構的優點是職責分離、易於擴展、易於測試。」

### 穩定性與 Debug（1分鐘）

> 「為了確保系統的穩定性，我實作了：
> 
> 1. **穩定性測試腳本**：驗證 embedding、chunking、retrieval 的一致性，確保相同輸入產生相同輸出。
> 
> 2. **Debug logging 系統**：每次 RAG 查詢都會自動記錄檢索過程和問答結果，方便快速定位問題。如果答案不準確，我可以查看檢索到的 chunks 和相似度分數，判斷問題是在 retrieval 階段還是生成階段。
> 
> 這體現了我的工程思維：不只是實現功能，還要確保系統的可觀察性和可調試性。」

### 技術選型（30秒）

> 「我選擇使用：
> - **FastAPI**：高性能的 Python Web 框架
> - **Ollama**：本地 LLM 和嵌入模型，不需要額外的 API key
> - **Cosine Similarity**：計算語意相似度，比歐氏距離更適合高維向量
> - **Top-K Retrieval**：只返回最相關的 k 個片段，避免引入噪音」

### 總結（30秒）

> 「這個專案展現了我的幾個能力：
> 1. 系統設計能力：從單一文件重構為分層架構
> 2. 工程思維：實作穩定性測試和 debug logging
> 3. 技術選型：選擇合適的技術棧解決問題
> 4. 可擴展性：設計抽象層，易於替換組件」

**總時長：約 4-5 分鐘**

## 🎮 2048 遊戲介紹

### 開場（30秒）

> 「我實現了一個 2048 遊戲，採用分層架構設計。雖然遊戲邏輯相對簡單，但我重點展示了如何組織代碼，讓它易於理解、測試和擴展。」

### 架構設計（1-2分鐘）

> 「我將遊戲設計為三層架構：
> 
> 1. **GameBoard 層**：負責遊戲板的狀態管理。包括初始化、隨機方塊生成、遊戲狀態檢查（是否可移動、是否已滿）。我特別加入了隨機種子支持，確保遊戲狀態可重現，這對於測試非常重要。
> 
> 2. **MoveHandler 層**：負責移動邏輯處理。我使用了一個巧妙的設計：通過旋轉遊戲板，將四個方向的移動統一轉換為向左移動，然後再還原。這樣只需要實現一次合併邏輯，大大減少了代碼重複，符合 DRY 原則。
> 
> 3. **ScoreCalculator 層**：負責得分計算和統計。使用靜態方法設計，不依賴實例狀態，便於測試。提供豐富的統計信息，可以支持未來的 AI 策略或性能分析。」

### 設計亮點（1分鐘）

> 「有幾個設計亮點：
> 
> 1. **統一移動處理**：通過旋轉技巧，將四方向移動統一為一個實現，減少代碼重複。
> 
> 2. **可重現性**：支持隨機種子參數，確保遊戲狀態可重現，這對於測試和調試非常重要。
> 
> 3. **職責分離**：每個模組職責單一，符合單一職責原則（SRP），易於測試和維護。」

### 總結（30秒）

> 「這個專案雖然簡單，但展現了我的：
> 1. 代碼組織能力：如何將簡單的邏輯組織成清晰的架構
> 2. 設計思維：使用設計模式和原則優化代碼
> 3. 工程實踐：考慮可測試性和可維護性」

**總時長：約 3-4 分鐘**

## 🤖 AI 預測模型介紹

### 開場（30秒）

> 「我開發了一個 AI 預測模型框架，採用分層架構設計，支持完整的機器學習流程：數據預處理、特徵提取、模型訓練和評估。」

### 架構設計（1-2分鐘）

> 「我將模型設計為三層架構：
> 
> 1. **DataPreprocessor 層**：負責數據預處理。我採用 scikit-learn 的 fit/transform 模式，確保訓練集和測試集使用相同的標準化參數，避免數據洩漏。同時提供異常值處理功能，使用 Z-score 方法識別和移除異常值。
> 
> 2. **FeatureExtractor 層**：負責特徵工程。提供多種特徵提取方法：基本統計特徵（均值、標準差、最大值、最小值、中位數）、時間序列特徵（移動平均、變化率）、交互特徵（特徵乘積、比值）。可以根據數據類型靈活組合，支持一維和多維數據。
> 
> 3. **Predictor 層**：負責模型訓練、預測和評估。使用策略模式，支持多種模型類型（線性回歸、隨機森林等），易於擴展。提供完整的評估指標（MSE、MAE、RMSE、R²），並且支持特徵重要性分析。」

### 設計亮點（1分鐘）

> 「有幾個設計亮點：
> 
> 1. **Fit/Transform 模式**：確保訓練和測試數據使用相同的標準化參數，避免數據洩漏。這是機器學習中的最佳實踐。
> 
> 2. **多種特徵提取方法**：可以根據數據類型靈活組合特徵。好的特徵工程往往比複雜的模型更重要。
> 
> 3. **完整的模型評估**：提供多種評估指標和特徵重要性分析，幫助理解模型性能和行為。」

### 總結（30秒）

> 「這個專案展現了我的：
> 1. 機器學習流程理解：從數據預處理到模型評估的完整流程
> 2. 工程實踐：使用 fit/transform 模式避免數據洩漏
> 3. 特徵工程能力：多種特徵提取方法，可以靈活組合
> 4. 模型評估能力：完整的評估指標和特徵重要性分析」

**總時長：約 3-4 分鐘**

## 🎯 通用面試技巧

### 1. 結構化表達
- 開場：簡要說明專案是什麼
- 架構：說明設計思路和技術選型
- 亮點：強調設計亮點和工程實踐
- 總結：總結展現的能力

### 2. 突出工程思維
- 不只是實現功能，還要考慮可維護性、可測試性、可擴展性
- 提到設計模式和原則（SRP、DRY、OCP 等）
- 強調穩定性測試和 debug 能力

### 3. 準備深度問題
- 如果讓你重做，你會怎麼改進？
- 你如何確保系統的穩定性？
- 你如何 debug 問題？
- 你如何評估模型的性能？

### 4. 展示學習能力
- 提到遇到的問題和解決方案
- 說明技術選型的考量
- 提到未來優化方向
